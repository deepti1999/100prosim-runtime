# §2.3 Literal paraphrase

**Source:** `260403_Portierung_Bestandsaufnahme.pdf` (H. Schmidt-Kanefendt, 2026-04-03), §2.3 "Datenmodell".
**Working text:** `docs/stakeholder/260403_Bestandsaufnahme_DE.md` lines 39–55 (German) / `260403_Bestandsaufnahme_EN.md` lines 40–56 (English).
**Purpose of this file:** capture exactly what §2.3 says, sentence by sentence, in literal English. **No interpretation, no commentary.** That happens in `DATA_MODEL_IMPORT_AUDIT.md` Step D / §1; this file is the source-of-truth quote.

The PDF section heading is `2.3 Datenmodell` (3 lines of opening proposal + two sub-sections `2.3.1 Nachvollziehbarkeit` and `2.3.2 Alternativ-Regionen` + a closing 1-line repetition of the proposal).

## §2.3 (opening, 1 sentence)

> **Vorschlag:** Schnittstelle zur Nutzung der bestehenden Excel-Datenmodell-Dateien anstelle des integrierten Datenmodells im aktuellen 100prosim-Web (Begründung im Folgenden).

**Literal:**
> **Proposal:** an interface for using the existing Excel data-model files instead of the integrated data model in the current 100prosim-Web (justification follows).

**Literal observations (no interpretation):**

- The whole §2.3 is framed as a **Vorschlag** (proposal), not a requirement.
- The object of the proposal is a **Schnittstelle** (interface) — not a specific implementation mechanism (the German does not say "import", "live binding", "sync", or "upload"; just "interface for using").
- The proposal targets the **existing Excel data-model files** (plural — "Dateien"). Not D.xlsx singular.
- The replacement is for the **integrated data model in the current 100prosim-Web** (i.e. the seed values + tables that ship inside the Django app today).
- A justification for the proposal follows in §2.3.1 and §2.3.2.

## §2.3.1 Nachvollziehbarkeit (Traceability) — 2 paragraphs

### Paragraph 1 (1 sentence + key claim about Excel)

> In 100prosim-Excel sind für jeden Parameter-Ansatz die Quellbezüge und getroffenen Annahmen einfach und direkt über Verlinkung nachvollziehbar. Das ist der entscheidende Unterschied zu einer Spielkonsole.

**Literal:**
> In 100prosim-Excel, for every parameter value the source references and the assumptions made are simply and directly traceable via hyperlinking. That is the decisive difference compared to a games console.

**Literal observations:**

- "Quellbezüge" = source references; "getroffene Annahmen" = the assumptions that were made.
- These are claimed to be "einfach und direkt über Verlinkung nachvollziehbar" — i.e. simply and directly traceable via hyperlinking. (The mechanism is hyperlinks inside the Excel workbook.)
- "Das ist der entscheidende Unterschied zu einer Spielkonsole" — i.e. without this traceability the tool degrades to a games-console (a black-box that produces numbers you cannot defend).

### Paragraph 2 (gap statement + admin update requirement)

> Im aktuellen 100prosim-Web sind Herkunft und Belastbarkeit der Parameter-Ansätze nicht nachvollziehbar. Dies ist aber Grundvoraussetzung für den Erkenntnisgewinn der Anwendenden durch kritische Auseinandersetzung zur Erweiterung und Festigung der eigenen Einschätzung. Dies gilt auch für die Parameter-Aktualisierung des Basis-Szenarios durch die Administrierenden, die zum Erhalt der Einsatztauglichkeit von Zeit zu Zeit erforderlich ist.

**Literal:**
> In the current 100prosim-Web, the **origin** and **defensibility** of parameter values are **not traceable**. This is, however, a **precondition** for the **insight gain** of users through **critical engagement**, for the **expansion** and **solidification** of their **own judgement**. The same applies to the **parameter update of the base scenario by the administrators**, which is required from time to time to keep the tool fit for deployment.

**Literal observations:**

- Two distinct **personas** are named:
  1. **Anwendende** (users): need traceability so they can critically engage with parameters and form their own judgement.
  2. **Administrierende** (administrators): need traceability so they can update parameters of the **base scenario** to keep the tool deployment-ready.
- The benefit ("Erkenntnisgewinn", "Festigung der eigenen Einschätzung") is **epistemological** — users learning by examining sources, not just receiving numbers.
- "Parameter-Aktualisierung des Basis-Szenarios … von Zeit zu Zeit erforderlich" — i.e. the BASE scenario itself needs periodic admin updates; this is a separate requirement from per-user scenario edits.
- "Herkunft" (origin) and "Belastbarkeit" (defensibility / robustness) are the two qualities to be made traceable; both are preconditions for the user-engagement and the admin-update purposes.

## §2.3.2 Alternativ-Regionen (Alternative regions) — 2 paragraphs

### Paragraph 1 (Excel state + architecture claim)

> Mit 100prosim-Excel wurde nicht nur die Region Deutschland als Basis-Szenario abgebildet, sondern auch verschiedene Bundesländer (<https://www.ernes.de/seite/422657/softwaretools.html>). Damit lassen sich die spezifischen Gegebenheiten der Region berücksichtigen. Ermöglicht wird dies durch die modulare Architektur, in der allein das Datenmodell sämtliche Regions-Spezifika enthält und so einfach austauschbar ist. Das Editieren des Datenmodells erfolgt hier in einer Excel-Datei (`D.xlsx`), spezielle Admin-Rechte sind nicht erforderlich.

**Literal:**
> With 100prosim-Excel, not only the region of Germany was modelled as a base scenario, but also various federal states (link: ernes.de/seite/422657/softwaretools.html). This allows the region-specific conditions to be reflected. This is enabled by the **modular architecture**, in which **only the data model contains all the region specifics** and is thus **easily swappable**. Editing the data model is done here in an Excel file (`D.xlsx`); **no special admin rights are required**.

**Literal observations:**

- Multiple regions exist in Excel-land: Germany + various Bundesländer (a URL to the public catalog is given).
- The architectural assertion: "**allein das Datenmodell** sämtliche Regions-Spezifika enthält und so einfach austauschbar ist" — i.e. all region-specifics live in the data model, and the data model is the swappable unit.
- File-level mechanism: **edit the .xlsx**. Filename convention `D.xlsx` for Germany; per-state files implied. **"Spezielle Admin-Rechte sind nicht erforderlich"** — i.e. anyone who can edit a spreadsheet can swap or update a region.

### Paragraph 2 (gap statement)

> Das aktuelle 100prosim-Web ist auf ein Deutschland-Datenmodell beschränkt. Erstellung und Zugriff auf Datenmodelle verschiedener Alternativ-Regionen wird nicht unterstützt. Dadurch ist eine regionsspezifische Anwendung, wie sie z. B. von verschiedenen Grünen-Landesarbeitsgemeinschaften intensiv genutzt wurde, nicht verfügbar.

**Literal:**
> The current 100prosim-Web is restricted to a Germany data model. **Creation of and access to data models of various alternative regions is not supported.** As a result, region-specific use — as for example was intensively used by various Green-party state working groups — is **not available**.

**Literal observations:**

- Web app is **Germany-only** today.
- Two missing capabilities, named explicitly:
  1. **Erstellung** (creation) of alternative-region data models.
  2. **Zugriff** (access) to alternative-region data models.
- **Concrete user group called out:** Grünen-Landesarbeitsgemeinschaften (Green party state working groups) — a real, in-the-field user base who lost capability when moving from Excel to Web.

## §2.3 closing (repetition of opening proposal, 1 sentence)

> **Vorschlag: Schnittstelle** zur Nutzung der **bestehenden Excel-Datenmodell-Dateien** anstelle des integrierten Datenmodells im aktuellen 100prosim-Web.

**Literal:**
> **Proposal: an interface** for using the **existing Excel data-model files** instead of the integrated data model in the current 100prosim-Web.

**Literal observations:**

- Identical to the opening sentence; the entire §2.3 is bracketed by this one proposal, with §2.3.1 + §2.3.2 acting as the two-pronged justification.
- The repetition emphasizes: the proposal is a **single architectural choice** (use Excel files as the data-model substrate), justified by the two sub-asks (traceability + multi-region).

## What §2.3 explicitly DOES say (literal extraction)

A flat list, no interpretation, of the asks/claims that appear verbatim in the German text:

1. The current integrated data model in 100prosim-Web should be **replaceable / supersedeable** by an interface that uses the existing Excel data-model files.
2. There are **multiple** Excel data-model files (plural "Dateien"); D.xlsx is named explicitly as the German one; per-Bundesland files exist (per the linked ernes.de catalog).
3. For **every parameter**, the **source reference** must be **traceable** (mechanism in Excel: hyperlinks).
4. For **every parameter**, the **assumption** that was made must be **traceable** (mechanism in Excel: cell comments / linked text).
5. **Users (Anwendende)** must be able to **critically engage** with parameter sources to gain insight and form their own judgement.
6. **Administrators (Administrierende)** must be able to **update parameters of the base scenario** periodically, to keep the tool deployment-ready, with the same traceability available.
7. The **architecture** must be **modular** such that the **data model alone** holds all region-specifics and is easily swappable.
8. Editing the data model must be possible **in an Excel file** without special admin rights.
9. **Creation** of alternative-region data models must be supported.
10. **Access** to alternative-region data models must be supported.
11. **Region-specific use** (named example: Bundesländer for Green-party state working groups) must be **available**.

## What §2.3 DOES NOT say (literal absence)

Things one might assume §2.3 mandates but that are NOT in the text:

- It does **not** specify the implementation mechanism (live read vs. import + sync vs. upload + parse). The word is **"Schnittstelle"** (interface), no further detail.
- It does **not** say the calculation engine, formula table, or any non-data-model layer must change. The scope is "Datenmodell" only.
- It does **not** require WS.xlsm, AH.xlsm, C.xlsx, MH.xlsx, or `_100prosim.xlsm` to be the substrate — these files are not named. The substrate it names is "Excel-Datenmodell-Dateien" plural and "`D.xlsx`" specifically (and its per-region siblings).
- It does **not** specify a UI surface for the source / assumption traceability beyond saying it must exist (no tooltip vs. side-panel vs. detail-page choice).
- It does **not** specify a tolerance, version-pin, or refresh cadence for the data import.
- It does **not** name a deadline, milestone, or owner.
- It does **not** declare 2.3 a hard requirement — the German is "Vorschlag" (proposal) twice.
- It does **not** specify whether per-user scenario edits should be retained, lost, or migrated when the base data model is updated.
- It does **not** specify what happens when an Excel file's structure changes between versions (schema evolution).
- It does **not** name a specific column in Excel as the "current value" — it just says parameters and their sources/assumptions are traceable.
