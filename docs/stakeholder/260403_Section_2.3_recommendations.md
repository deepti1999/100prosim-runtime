## Per-decision detail (only where the table is too compact)

### D1 — Provenance schema (additive)

**Why not extend `quelle` / `source` / `notes`?**

- `LandUse.quelle` carries strings like `"D.1.64"`. That is a paragraph-style code from the original D.xlsx structure (`D.<sheet>.<paragraph>`), **not** a row-into-D.xlsx pointer. Reusing this column to hold a URL would mix two semantics in the same field; downstream code that already reads `quelle` (grep `simulator/` for `\bquelle\b`) would receive unexpected URL text.
- `RenewableData.source` carries strings like `"solar_energy.csv"` — an old import-source identifier from the seed pipeline. Same problem.
- `notes` is empty everywhere (0 / 420). Could in principle absorb assumption text, but mixing a stakeholder-facing assumption note with developer-facing notes is the kind of conflation that bites later.

**Concrete schema:**

```python
# additive migration, no removal
source_url        = models.URLField(null=True, blank=True)
source_ref        = models.CharField(max_length=64, null=True, blank=True)  # e.g. "9.224" — D.xlsx Quellen index
notes_assumption  = models.TextField(null=True, blank=True)
origin            = models.CharField(max_length=16, choices=ORIGIN_CHOICES, default='internal')
```

`ORIGIN_CHOICES`: `d_xlsx`, `derived` (computed from a d_xlsx parent), `internal` (ours-only).

### D4 — UI surface (the one Pascal might want to opine on)

Three viable options:

1. **Info icon, click-to-popover** (recommended). One small clickable indicator per row; click opens a Bootstrap popover (we already use Bootstrap) with linked source URL + assumption text. Keyboard accessible (Tab + Enter); screen-reader-friendly with `aria-describedby`; trivially Playwright-testable (`page.click` + assert popover content).
2. **Hover-tooltip** (Excel-faithful). Mimics Excel's cell-comment hover behaviour. More immediately discoverable but: (a) inaccessible without explicit `tabindex` work, (b) Playwright hover-then-snapshot is flakier than click, (c) doesn't render on touch devices.
3. **Side panel** (slide-in drawer). Higher ceremony, eats horizontal space, useful for richer content but our content is just URL + ~1 paragraph; overkill.

If Pascal prefers option 2 we can ship that — the backend work (SR-002/003 columns) is identical; only the template + JS bind differs. Either choice is reversible without re-importing.

### D6 — Origin classification details

Per Step C s_xlsx_map evidence, the unmatched rows are predominantly:

- **Category headers** (RenewableData `1 Solarenergie`, `1.1.1 Solarthermie`, `1.1.2 Solarstrom`, `2.1 Onshore-Windstrom`, etc.) — exist in our DB as parents; no value because they aggregate children. `origin='internal'` with rationale "category header — value is sum of children".
- **Aggregations** (VerbrauchData `4.0 Mobile Anwendungen (MA)`, `9 Grundstoff-Synthetisierung`) — same pattern. `origin='internal'` with rationale "aggregation row".
- **Deeper splits** (RenewableData `10.4.3 davon Strom`, `10.6.2 davon Strom`) — our hierarchy goes one level deeper than D.xlsx. `origin='derived'` with rationale "deeper split of parent code <X>".
- **One literal placeholder row** (RenewableData `5` with empty label) — should be `origin='internal'` with rationale "TBD placeholder; investigate before next import".

The import tool emits a one-CSV report `data/import/orphan_classification.csv` listing each row + assigned origin + rationale. Pascal reviews once per import; thereafter classification is deterministic.
