## 0. The 30-second answer

§2.3 asks for **traceability + multi-region**, NOT for a value import. The
values are already in our DB (with 78 % matching _S.xlsx 1:1 by label and
value at HIGH confidence; 92.8 % at MED-or-better). What is missing is:

1. **Source URLs** — the 86 hyperlinks on `D.xlsx!9.Quellen` are not in the DB.
2. **Assumption notes** — the 747 per-cell comments on `D.xlsx!1.` are not in the DB.
3. **Region swap** — there is no first-class `Region` concept (Germany is implicit).
4. **Admin parameter editability without code** — currently no UI for an admin to update the base scenario.

The previous v1 audit (`DATA_MODEL_IMPORT_AUDIT.md` pre-revision) framed §2.3 as
"import D.xlsx values into the DB". That framing is wrong. The values are not
the gap; the **provenance + region + admin-edit layer** is.

## 3. Why the value-import framing was wrong

Direct quotes from the v1 audit (`DATA_MODEL_IMPORT_AUDIT.md` pre-revision):

> "the 420 parameters that feed the simulator today are hardcoded in
> `seed/sqlite_seed.json` with no visible provenance. The stakeholder wants the
> D.xlsx (Germany data model, 17 sheets, 675 parameter rows, 86 source
> hyperlinks, 747 assumption comments) to be the **authoritative source** —
> every parameter editable per-scenario but traceable back to its D.xlsx row,
> source URL, and assumption comment."

This conflates two different things:

1. ✅ "Every parameter traceable back to its **source URL + assumption comment**" — yes, this is L3 + L4, that's the gap.
2. ❌ "D.xlsx as the **authoritative source** … every parameter [needs] re-importing" — no, the values are already in the DB. _S.xlsx mapping evidence (Step C) shows 78 % HIGH-confidence value match (matched by label AND status AND ziel) and only 1.2 % NONE matches. Re-importing values would re-introduce a worse-quality copy of what we already have.

The Step C mapping numbers (78 % / 14 % / 6 % / 1 %) are the empirical
counter-argument: **the values are not the gap.** They might drift slightly
between a stakeholder Excel update and our DB (Pascal can re-sync deliberately
when that happens), but day-zero we already have them. The gap is the
**provenance metadata** that travels with the value, and the **region + admin**
infrastructure around the model.

## 4. What §2.3 actually wants — restated as 4 buckets

| # | Bucket | What it means concretely | Maps to literal asks |
|---|---|---|---|
| **B1** | **Provenance import** | Pull the 86 source URLs from `D.xlsx!9.Quellen` and the 747 assumption comments from `D.xlsx!1.` into per-parameter `source_url` + `notes_assumption` fields on the 4 models. Expose in UI as a tooltip / info icon. | L3, L4, L5 |
| **B2** | **Region first-class** | Add `Region` model (`code`, `display_name`, `active`); link parameters to regions; allow swapping the active region. Initial release: Germany only, but the schema is region-aware. | L7, L9, L10, L11 |
| **B3** | **Admin re-import flow** | Management command `manage.py import_excel_datenmodell <region>.xlsx` that reads a stakeholder-provided Excel file (D.xlsx or per-region equivalent) and updates the **base** scenario rows for that region. Idempotent + diff-on-rerun. Surfaced in admin via a small UI. | L6, L8 |
| **B4** | **Optional value re-sync** | When the stakeholder ships an updated D.xlsx with revised values, B3's import is the route. Note: this is **not a one-time bulk import of all 420 rows** — it is a selective update that diff-and-confirms changes against the existing DB, so the per-user workspace data stays intact. | L1 (interpreted as "interface to use" not "replace contents") |

**Important relationship:** B3 is the mechanism that satisfies B1 (provenance
arrives via the import) and B4 (value re-sync via the import). B2 is
orthogonal but lives in the same model layer, so it ships alongside.

## 5. What's NOT in §2.3 scope (the boundary)

To prevent scope creep, things to **defer or reject** as part of §2.3 work:

| Out of scope | Why |
|---|---|
| Replacing `calculation_engine/` with Excel-runtime calls | The math is already ported. Reading Excel per request would tank Heroku perf (PDF §2.2 acid test is already 20× slower than Excel, can't make it worse). PDF §2.3 doesn't ask for this. |
| Reading any workbook at request time | Same. "Schnittstelle" allows import; it doesn't require live binding. Confirmed by literal paraphrase §"What §2.3 DOES NOT say". |
| Replacing `Formula` table or formula engine | Out of §2.3 scope. The 760 formulas are ours; D.xlsx doesn't supply them, WS.xlsm does (and that's already ported). |

| AH.xlsm / C.xlsx / MH.xlsx / `_100prosim.xlsm` / trace logs | Not Datenmodell files. Out of scope. |
| Per-user override behaviour change | Orthogonal. T14–T18 covers this. |
| Renaming any cell / code / label | **Forbidden** by CLAUDE.md "Stakeholder requirements" #1. |

## 6. Decision points that ARE blocking (and the recommended call)

These are the choices Pascal needs to make before implementation can start. The
Step E rewrite of `DATA_MODEL_IMPORT_AUDIT.md` will surface these in §9. The
v1 audit had **10 decision points**; most were premature scoping for a value
import that isn't the right framing. Real blockers:

| # | Question | Recommendation | Reasoning |
|---|---|---|---|
| D1 | **Provenance schema** — extend existing `source` / `quelle` / `notes` columns OR add new `source_url` + `notes_assumption` + `origin` columns? | **Extend `notes` for assumption text; add `source_url` + `source_ref` columns; add `origin` enum.** `source` / `quelle` already exist with content (LandUse.quelle has D-codes); extending them risks misinterpretation. | Backwards-compatible; additive; preserves the partial L3 work |
| D2 | **Region model** — first-class table OR a settings constant (e.g. `ACTIVE_REGION = 'DE'`)? | **First-class table**, schema like `(code: str pk, display_name: str, active: bool, datenmodell_excel_hash: str)`. Even if Day 1 only has one row (`DE`), the schema makes B2 cheap to extend later. | Schema cost is tiny (20 LOC migration); avoids future rewrite when ErnES sends BB.xlsx |
| D3 | **Import command behaviour** when D.xlsx is missing or unparseable? | **Fail loud, no silent fallback.** Refuse to update if the file or column manifest changed in a way the import tool doesn't recognize. Output diff for human review. | Mitigates RISK-06 (D.xlsx column reshape between stakeholder versions) |
| D4 | **Source-URL UI surface** — tooltip on hover, info-icon click, side panel, detail page? | **Info icon (i) per row, click-to-popover.** Cheap to render, doesn't crowd existing UI, accessible, easy to test in Playwright. | Doesn't conflict with existing UI; cheap V4/V5 testing |
| D5 | **D.xlsx file pinning** — do we commit a hash manifest so we know which D.xlsx version was last imported? | **Yes** — a `data/mapping/d_xlsx.manifest.json` with file hash + import timestamp + import-tool version. The .xlsx file itself stays gitignored. | Audit trail; supports re-sync conversations with Schmidt-Kanefendt |
| D6 | **For LOW-confidence DB rows** (the 5 NONE + 21 LABEL_ONLY in renewabledata, 4 LABEL_ONLY in verbrauchdata) — classify as `origin='internal'` OR ask Schmidt-Kanefendt to add them to D.xlsx? | **Classify as `origin='internal'`.** They are mostly category headers (Solarenergie, Solarthermie, Solarstrom — categorical labels with no numeric value) or deeper hierarchy than D.xlsx splits. Asking the stakeholder to extend D.xlsx delays the audit-close indefinitely. | Pragmatic; can be promoted later if stakeholder asks |
| D7 | **Phase-A scope** — schema + provenance import only, OR include the UI tooltip in Phase A? | **Schema + provenance + tooltip in one phase.** All three are tightly coupled and small (~3 days). Splitting forces two V5 Heroku verifications instead of one. | Saves V5 cycles (~$0.10 each); keeps visible value per phase |
| D8 | **Region first-class — Phase A or Phase B?** | **Phase B.** Phase A focuses on provenance for the existing single-region data. Region + import-from-Excel ships in Phase B; doesn't block the user-visible improvement. | Lets us validate provenance UX without coupling to region migration risk |

The 10 decision points in v1 §9 included items (column-naming, mapping CSV
location, value-sync policy, Heroku cadence, commit boundaries, etc.) that
either don't apply (no value sync needed) or are answered above (D1–D8).

## 7. The gap remaining — punch list

What concretely needs to be built (subject to Pascal sign-off on D1–D8):

### Phase A — Provenance + tooltip + admin import (single-region)
1. Schema migration on 4 parameter models: add `source_url`, `source_ref`, `notes_assumption`, `origin` columns. Backfill `origin = 'd_xlsx'` for matched rows, `'internal'` for the rest.
2. `manage.py import_excel_provenance D.xlsx` — reads `D.xlsx!9.Quellen` for URLs and `D.xlsx!1.` cell comments for assumption text; matches to our DB rows by label using the s_xlsx_map_*.csv evidence; populates the provenance fields. Does NOT touch the value columns.
3. UI tooltip / info-icon on each row of `/landuse/`, `/renewable/`, `/verbrauch/`, `/gebaeudewarme/` showing source + assumption.
4. V2 tests: import idempotency, provenance population, UI tooltip presence.
5. V4 / V5 Playwright: each parameter page renders the icon + popover correctly; existing scenarios A / C / D unaffected.

### Phase B — Region first-class + import command + Bundesländer-ready
1. Schema migration: `Region` model + region FK on the 4 parameter models (default `DE`).
2. `manage.py import_excel_datenmodell <region_code> <file.xlsx>` — reads a Datenmodell .xlsx (D.xlsx shape), inserts/updates rows tagged with the region; uses Step C s_xlsx_map_*.csv to know which sheet/row maps to which DB row.
3. Admin UI: file upload + region-select on the admin landing.
4. UI region selector (top-bar) — switches the **read** scope. Default DE so existing users see no change.
5. V2 tests: per-region row isolation; region switcher persistence; no cross-region leak.
6. V4 / V5: region switcher works on live Heroku; Bundesländer file upload accepts a sample file (use a copy of D.xlsx for first round; per-state files arrive from ErnES later).

### Phase C — (defer until B + provenance hold up)
- Per-region provenance (extend B1 to be region-aware).
- Per-region UI labels (currently German-only, may need region naming).
- Diff-and-review re-sync conversations with Schmidt-Kanefendt as D.xlsx versions land.

### Out of scope
- Phase D from v1 audit (separate UI provenance phase) — folded into Phase A.
- Phase B value-sync policy from v1 audit — replaced by the import command.
- All 10 v1 decision points — most resolved by reframing.

## 8. Risk register update — what changes vs. v1 audit's 10 risks

The v1 audit's RISK-01 ("Importing a D.xlsx numeric value silently drifts a
scenario golden") was the **biggest concern in the wrong framing** because
value-sync was central. **In the corrected framing, RISK-01 mostly disappears:**
Phase A doesn't touch values. Phase B introduces values only for **new regions**
(no override of existing DE rows unless Pascal explicitly imports a fresh DE
file). Each risk re-evaluated:

| ID | v1 wording | New status | Why |
|---|---|---|---|
| RISK-01 | Value-import drifts scenario golden | **Drops to LOW** | Phase A is provenance-only; Phase B adds new regions. Refresh of DE values is a separate, opt-in admin operation. |
| RISK-02 | Ambiguous one-to-many label mappings auto-picked wrong | **Drops to MEDIUM** | Step C s_xlsx_map confidence is 78 % HIGH; 22 % MED-or-below. Hand-curate only the MED + LOW rows (90 rows total, ~30 min). |
| RISK-03 | Code/label rename during import | **UNCHANGED — Critical** | Linter test still required. |
| RISK-04 | Import collides with `ensure_user_workspace_data()` | **UNCHANGED — Medium** | Maintenance-mode flag still required for re-import operations. |
| RISK-05 | Per-user overrides overwritten | **UNCHANGED — Critical** | Import touches `owner=NULL` base rows only; user workspace tables are out of scope. |
| RISK-06 | D.xlsx layout change between revisions | **UNCHANGED — Medium** | Pinned manifest + fail-loud import. |
| RISK-07 | Orphan classification | **UNCHANGED — Low** | `origin='internal'` for the ~30 unmatched rows handles it. |
| RISK-08 | Idempotency | **UNCHANGED — Medium** | Content-hash per row; second run = no-op. |
| RISK-09 | Jahresstrom (T54 D1–D4c) numbers move after Phase B | **Drops to LOW** | Phase B only adds new regions; existing DE values untouched in Phase A or B. Track 1 D1–D4c outputs are stable. |
| RISK-10 | Provenance column collisions with stakeholder-frozen names | **UNCHANGED — Low** | Add new `source_url` + `notes_assumption` + `origin` columns; don't rename `source` / `quelle` / `notes`. |

**Net:** the highest-severity risk (RISK-01) drops from High/High to Low/Low
because we are no longer doing value import in Phase A.
