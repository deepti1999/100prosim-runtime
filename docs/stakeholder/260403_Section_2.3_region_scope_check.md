## Verdict (TL;DR) — original 2026-04-23 reading

**The PDF leans toward "≥2 regions actually working." Phase B is operationally incomplete; Phase C is mandatory if we read §2.3.2 ¶2 literally.**

But the §2.3 framing is "**Vorschlag**" (proposal) not "**Pflicht**" (mandate), so a stakeholder check-in with Schmidt-Kanefendt is the cleanest unblock. Two coherent stakeholder positions are defensible:

- **"Architectural support suffices"** → Phase B done; per-Bundesland data is operational follow-up. Close §2.3.
- **"At least one Bundesland must demonstrate end-to-end use"** → Phase B incomplete; open Phase C now.

The literal text leans toward the second reading. Recommendation: open Phase C **OR** explicitly ask Schmidt-Kanefendt to ratify the first reading and close §2.3.

## What §2.3 literally says (extracted verbatim)

From `260403_Section_2.3_literal.md` (Step A) and the German source at `260403_Bestandsaufnahme_DE.md` lines 39–55.

### §2.3.2 ¶2 — the gap statement

> Das aktuelle 100prosim-Web ist auf ein Deutschland-Datenmodell beschränkt. **Erstellung und Zugriff auf Datenmodelle verschiedener Alternativ-Regionen wird nicht unterstützt.** Dadurch ist eine regionsspezifische Anwendung, wie sie z. B. von verschiedenen Grünen-Landesarbeitsgemeinschaften intensiv genutzt wurde, **nicht verfügbar**.

**Literal English:**

> The current 100prosim-Web is restricted to a Germany data model. **Creation of and access to data models of various alternative regions is not supported.** As a result, region-specific use — as for example was intensively used by various Green-party state working groups — is **not available**.

### The two-prong proposal (opening + closing of §2.3)

> **Vorschlag:** Schnittstelle zur Nutzung der **bestehenden Excel-Datenmodell-Dateien** anstelle des integrierten Datenmodells im aktuellen 100prosim-Web.

**Literal English:**

> **Proposal:** an interface for using the **existing Excel data-model files** instead of the integrated data model.

The German is "Dateien" (plural) and per-Bundesländer files are explicitly named with a public link (ernes.de/seite/422657/softwaretools.html).

### The atomic asks L1–L11 in scope here

(Full list in `260403_Section_2.3_literal.md` §"What §2.3 explicitly DOES say".)

The three asks that bear directly on this question:

| ID | Literal ask |
|---|---|
| L7 | The architecture must be **modular** such that the **data model alone** holds all region-specifics and is easily swappable. |
| L9 | **Erstellung** (creation) of alternative-region data models must be supported. |
| L10 | **Zugriff** (access) to alternative-region data models must be supported. |
| L11 | **Region-specific use** (named example: Bundesländer for Green-party state working groups) must be **available**. |

L7 is architectural. L9, L10, L11 are operational.

### Operationally satisfied **for DE only**

Phase B did NOT exercise any non-DE region. The 5 things that would need to happen for any Bundesländer to actually work:

1. **Region row creation** — works (`Region.objects.create(code='BB', ...)`).
2. **Per-region parameter row creation** — **does NOT work.** No tool creates 420 BB base rows with BB-specific values. The `import_excel_provenance --region=BB` command only **updates** existing rows; verified by reading `_compute_diff` — it iterates `model.all_objects.filter(owner__isnull=True, region=region)` and writes only `source_url`, `notes_assumption`, `origin`. **It never creates rows.**
3. **Per-region D4a/D4b values** — works if `Region.installed_pmax_*` is populated at row-create time, but no tool reads those values from a Bundesländer Excel file. Manual entry only.
4. **BalanceJob worker per region** — TODO comment in `simulator/balance_jobs.py:37` notes the worker hardcodes DE. A BB user clicking Balance would run against DE workspace, not BB.
5. **Scenario / Baseline snapshots per region** — TODO in `simulator/baseline_api.py` excludes `region` FK from per-row payload. Restored snapshots default to DE regardless of which region the user was active in when they saved. Cross-region snapshot leakage at restore time.

### The 4 TODO comments

(Listed here for completeness; full text in `DATA_MODEL_IMPORT_AUDIT.md` §0b "Phase 2 follow-ups".)

| File | Type | What flips when ≥2 regions exist |
|---|---|---|
| `simulator/balance_jobs.py:37` | TODO | `BalanceJob.payload` must carry `region_code`; worker wraps dispatch in `region_scope(...)`. **Without this, a BB user clicking Balance gets DE balance.** |
| `simulator/models.py:1537` | Note | `GebaeudewaermeData.code` must swap from global `unique=True` to `UniqueConstraint(['region','code'])`. **Without this, BB cannot have a row sharing a code with DE.** |
| `simulator/baseline_api.py` (`_serialize_model_rows` / `_restore_model_rows`) | Note | Scenario / baseline snapshot payload needs a top-level `region_code`; restore must reapply that region. **Without this, BB scenarios round-trip as DE scenarios.** |
| `simulator/workspace_service.py:120` | Note | WSData (365-day series) needs a per-region scope or a per-(user, region) clone. **Without this, BB shares the user's DE timeseries.** |

These are **not optional polish** the moment a non-DE region exists. They're the difference between "BB works correctly" and "BB silently corrupts".

## What "Phase C" would have to deliver

If Pascal opens Phase C in response to the literal reading, it's a small but real piece of work:

1. **Value-ingest tool for new regions.** Either:
   - New command `manage.py seed_region --region=<code> --from=<xlsx>` that bulk-creates 420 base rows reading values from the region-specific Excel file; OR
   - Extend `import_excel_provenance` with a `--create-rows` flag that also writes values when the row doesn't exist for that region.

2. **Flip the 4 TODOs.** Each is a localized change:
   - `BalanceJob.payload` carries `region_code`; worker wraps `region_scope`.
   - `GebaeudewaermeData.code` constraint becomes `(region, code)`.
   - Scenario / baseline serializer adds top-level `region_code`; restore re-binds.
   - `WSData` either gains a region FK and per-region clone helper, or gets explicit "shared across regions" docs.

3. **Smoke-test end-to-end with one non-DE region.** Even synthetic — e.g. `BBSYNTH` with values 0.5× DE — proves the full flow works (dropdown switch → workspace clone → balance run → snapshot save/restore → diagram with BB pmax constants).

4. **(Optional) GUI for L8.** A `/admin/data-import/` form that uploads a Bundesländer Excel + creates the Region + ingests values + previews diff. Closes L8 properly.

Estimate: **2–3 days** if Pascal already has one Bundesländer Excel file (else the smoke-test uses synthetic values). One V5 Heroku cycle.

## Recommendation

**Open Phase C — small, ~2-3 days, single V5 Heroku cycle.**

Reasoning:

1. The literal §2.3.2 ¶2 names the gap as both "Erstellung und Zugriff" being unsupported and the named user group ("Grünen-Landesarbeitsgemeinschaften") having no available region-specific use. Phase B fixes neither for any actual Bundesland.
2. The 4 TODOs are not deferable polish — they are bugs that activate the moment a 2nd region exists. Leaving them open means any Bundesländer ingest in the future will hit silent breakage.
3. Phase C is bounded: 1 new management command (or extend the existing one), 4 TODO-flips, 1 end-to-end smoke test. Easier to do now (Phase B context fresh) than later (context decayed).
4. Even with synthetic data for the smoke test (no real BB.xlsx in the bundle), Phase C *proves* the architecture works end-to-end. That's a stronger position to hand off to ErnES than "scaffolding plus 4 TODOs".

**If Pascal disagrees and wants to defer:** open a Schmidt-Kanefendt question explicitly: *"Reicht uns die architektonische Unterstützung von Alternativ-Regionen, oder müssen wir mindestens eine Region (z. B. Brandenburg) konkret befüllen, um §2.3.2 als erfüllt zu gelten?"* Then close §2.3 with whichever answer comes back.
